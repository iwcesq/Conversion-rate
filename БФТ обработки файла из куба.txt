Требования к бизнесу: Подготовка данных о продажах и трафике

Проект: LFL Regression Analysis  
Цель: Получить очищенный, готовый к анализу набор данных путём фильтрации системного шума, ошибок телеметрии и нереалистичных выбросов.

---

1. Правила целостности данных и фильтрации

Пайплайн данных должен применять следующие фильтры в указанном порядке.

1.1 Логическая согласованность — Призрачные продажи

Требование: Удалять записи, где зафиксирована выручка/чек без соответствующего трафика посетителей.  
Логика: Удалять строки, где Receipts > 0 И (People == 0 или People IS NULL).  
Обоснование: Предотвращает искажение метрик конверсии транзакциями, которые не связаны с отслеживаемыми посетителями (например, телефонные заказы или корректировки системы).

1.2 Удаление системного шума — Час 0

Требование: Исключать все данные с Hour == 0 (полночь).  
Логика: Фильтровать Hour == 0.  
Обоснование: Данные за полночь часто содержат пакетно обработанные транзакции, сбросы системы или логи обслуживания, которые не отражают реального поведения клиентов.

1.3 Временные ограничения — Будущие даты

Требование: Исключать записи с датой в будущем.  
Логика: Date < 2025-11-22.  
Обоснование: Устраняет плейсхолдеры или тестовые записи, случайно внесённые в продуктивную базу данных.

1.4 Порог правдоподобия — Коэффициент конверсии

Требование: Ограничить данные реалистичными диапазонами конверсии.  
Логика: Оставлять только записи, где 0.09 < Conversion <= 1.00 (9%–100%).  
Обоснование:  
- Значения выше 100% математически невозможны (чеков больше, чем людей).  
- Значения ниже 9% на основании исторической работы магазинов указывают на технические неисправности (например, сбой счётчика посетителей).

---

2. Трансформация данных и типизация

Для поддержки регрессионного моделирования итоговый набор должен соответствовать следующей схеме:

| Column | Data Type | Note |
|---|---|---|
| Date | Date | Стандартный ISO формат YYYY-MM-DD |
| Hour | Categorical / Factor | Рассматривать как дискретную категорию, не как непрерывное целое |
| Weekday | Categorical / String | Вычисляется из Date, например Monday; Tuesday |
| Receipts | Decimal / Float | Целевая переменная |
| People | Integer | Основной предиктор |
| Conversion | Decimal / Float | Вычисляется как Receipts / People |

> Sources: none

---

3. Контроль качества и отчётность

После выполнения пайплайна система должна формировать сводный отчёт, включающий:

- Input Row Count: общее число записей до очистки.  
- Output Row Count: общее число записей после очистки.  
- Exception Log: количество строк, удалённых по каждому правилу фильтрации (например, «500 строк удалено по правилу Future Date»).  
- Null Check: в итоговом наборе должно быть 0 NULL в полях Receipts, People и Conversion.

4. Место сохранения результата

Формат: CSV или выделённая аналитическая таблица в хранилище данных, например factsalescleaned_lfl.  
Частота: указать — например, ежедневная или одноразовый экспорт.

Рекомендации по взаимодействию с IT или командой Data Engineering

- Объясняйте почему: Укажите, что 9% — это минимально правдоподобный уровень конверсии; это не произвольное число, а бизнес-порог, основанный на исторических данных.  
- Просите не только CSV, а View: Если используется SQL, попросите создать Database View. Тогда новые данные будут автоматически очищаться по заданным правилам по мере поступления, и не потребуется вручную запускать R-скрипт для очистки.
